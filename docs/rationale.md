# 设计动机

很多人同时使用 Codex、Hermes、OpenClaw、Claude Code、Cursor 和自定义脚本。每个工具都能记住一点事情，但这些记忆通常散在不同会话、不同电脑和不同格式里。时间一长，常见的问题就会出现：同一偏好被反复记录，旧决定覆盖新决定，某台设备的事实被误用到另一台设备，Agent 也说不清它为什么会“想起”某句话。

Agent Memory Gateway 把共享记忆独立出来，让它成为所有运行时都能连接的一层服务。它保存长期、有来源、可追溯的信息，同时把完整会话、项目文件和即时工具状态留在各自原来的位置。

## 为什么要有 Gateway 和 Sidecar

共享记忆需要统一的身份、权限、审计和冲突处理；客户端又需要在设备离线时保留自己的队列和凭据。因此系统把职责分开：

```text
本机 Sidecar：接入、凭据、加密离线队列和缓存
中心 Gateway：身份确认、工作区授权、记忆裁决和审计
```

Gateway 在检索前先筛掉无权访问的记录，向长期记忆后端只传递已授权的引用。Sidecar 则让本机多个 Agent 共用一份队列和本地接口，避免每个 Agent 各自保存一套同步状态。

## Markdown 和 Obsidian 适合放在哪里

Markdown 很适合人阅读、整理知识和导出归档，Obsidian 也很适合承担这一层。但多 Agent 同时写入时，还需要幂等、冲突处理、作用域、审计和稳定的检索字段；这些职责放进文件同步会让边界变得模糊。

因此，Markdown 可以作为导入来源和导出形式。共享库本身由 Gateway 管理事实状态，必要时再把审核后的内容整理回人类可读的文档。

## 为什么使用独立的共享记忆层

Letta、LangGraph 等工具各自解决 Agent 运行时或工作流编排的问题。共享记忆层需要同时服务多个运行时，并保持一致的工作区权限、事件账本和审核规则。

这种分层让 Codex、Hermes、OpenClaw 和其他支持 MCP 或 HTTP 的客户端可以使用同一套长期记忆，同时继续保留各自的会话、插件和工作方式。
