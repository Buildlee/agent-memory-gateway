# 日常运维与恢复

适用于已部署 Gateway 和本机 Sidecar 的管理员。所有检查默认只读——不会重放事件、清理 outbox、修改审核结果或删除数据。

---

## 🔍 运行状态检查

设备上安装了项目且 Sidecar 正在运行时执行：

```powershell
.\scripts\check-admin-health.ps1 `
  -AgentInstallationId "your-admin-agent-id" `
  -DefaultWorkspace "your-workspace-id"
```

脚本从本机受保护的 Sidecar key 文件读取回环鉴权 key，再调用 Sidecar。不会打印 key、Gateway 令牌、刷新凭据、数据库连接串或记忆正文。

在克隆的源码目录中运行且尚未安装命令入口时，脚本只对当前仓库 `src` 目录做进程内回退——不安装依赖、不改全局环境、不读 Gateway 凭据。正式安装后的行为不变。

输出 JSON：

- `ok: true` — worker 心跳正常，没有待重试事件和未处理死信
- `WORKER_HEARTBEAT_STALE` — worker 最近一次心跳超阈值。检查 Gateway 和 worker 的健康状态与日志
- `RETRYABLE_EVENTS_PRESENT` — 有事件等待重试。确认网络、数据库和后端依赖是否可用
- `DEAD_LETTERS_PRESENT` — 有事件已停止自动重试，需人工判断

退出码 `0` 正常，`1` 发现运行问题，`2` 表示 Sidecar、工作区配置或管理授权不可用。Windows 计划任务、监控平台或通知脚本可据此发提醒；不要把检查输出连同环境变量写进公共日志。

## 🌐 管理页：中枢优先，本机备用

正式环境应把管理页部署在 Gateway 所在的中枢环境，通过 HTTPS `/admin` 统一访问；本机 `127.0.0.1` 管理页保留给离线维护和单机排查。两种入口都需要已登记的 `memory.manage` 身份，浏览器都不会直接访问 Gateway 或数据库。

中枢入口的初始化、浏览器首次授权、固定地址访问、容器边界和验收步骤见[中枢管理页](central-admin.md)。它使用独立管理 Sidecar，并且只把 Caddy `/admin` 暴露给已在内网或 VPN 边界内的浏览器。完成一次授权后，浏览器可在 30 天有效期内直接打开固定地址；容器重启不会要求重新运行打开脚本。

本机备用入口仍可按下面方式启动：

```powershell
.\scripts\start-admin-console.ps1 `
  -AgentInstallationId "your-admin-agent-id" `
  -DefaultWorkspace "your-workspace-id"
```

脚本读取本机 Sidecar key 文件，只设置访问 Sidecar 所需的环境变量，清掉继承的 Gateway 令牌、刷新凭据、设备 ID 和 CA 配置。启动后打印 `http://127.0.0.1:<port>/?session=...` 地址——该地址仅用于首次换取本机会话 Cookie，不可重复使用。中枢入口不在终端打印链接，而是由打开脚本直接交给浏览器。

管理页、Sidecar 和 Gateway 必须使用同一版本。若页面提示 `LOCAL_METHOD_UNSUPPORTED`，说明 Sidecar 仍是旧进程——先完成版本更新，在维护窗口重启这台设备的 Sidecar，然后重新运行健康检查。不要通过浏览器直连 Gateway 来绕过。

Windows Sidecar 启动脚本优先从当前发布副本的 `src` 目录加载代码。这样更新源码并重启 Sidecar 时，无需替换可能被 Codex 或 Hermes MCP 占用的 `.exe` 文件。发布副本必须保持在已验证版本；不要把试验目录设为计划任务的工作目录。若项目目录没有 `src`，脚本继续使用已安装包。

页面六个入口：

- **概览** — 待审核、待重试、死信、活跃设备、健康检查和近期活动；四个状态卡可直接进入对应页面
- **记忆** — 按关键词检索当前工作区内、当前 Agent 已获授权的记忆；不会新建、删除或批量改写
- **审核** — 确认原文、编辑后确认、保留双方、取代冲突、拒绝和归档
- **设备与权限** — 设备和 Agent 名称、类型、状态、工作区绑定、能力、最近状态与认证版本；可以修改当前工作区能力、撤销 Agent 或撤销设备，技术标识默认折叠，不显示公钥或凭据
- **运行** — 待重试、未处理死信和只读恢复检查
- **活动** — 近期管理与审核记录，可按结果和关键词筛选，并显示来源设备与 Agent；不显示正文或敏感详情

审核、权限修改和撤销操作都要在页面上再次确认。请求携带当前 revision、授权 epoch 或期望能力集，以及幂等键；数据变化后不会静默覆盖。当前管理端不能撤销自身，也不能移除自身的 `memory.manage`。管理页不提供删除、批量重放、清理 outbox 或直接改数据库的按钮——这些动作需要单独的受控流程。

## 🔧 只读 MCP 排查工具

拥有 `memory.manage` 的 Agent 可使用以下只读 MCP 工具：

- `memory_admin_overview` — 审核、重试、死信和设备数量及 worker 心跳时间
- `memory_admin_dead_letters` — 未处理死信的 ID、错误码、错误类别和创建时间
- `memory_admin_audit` — 近期操作的时间、操作者、结果码和 trace ID
- `memory_admin_devices` — 设备、Agent、工作区绑定和权限状态

这些工具不返回设备公钥、凭据、`details_json`、事件正文或密文。排查时先用 trace ID 和错误码关联受保护环境的服务日志；不要把日志中的令牌、连接串或用户内容复制到 issue、聊天记录或 Git。

## ⏳ 恢复顺序

**心跳过期时** — 先确认 HTTPS 入口和 Gateway 健康检查，再看 worker 日志中是否有数据库连接、迁移版本或后端依赖错误。服务恢复后重新运行只读检查，确认心跳更新时间已推进。

**待重试事件** — 通常 worker 会按既有退避策略自动处理。不要手工重复提交原始写入请求；先等依赖恢复，再观察数量是否下降。持续增长时暂停新的变更操作，保留审计和事件账本，定位首个错误码。

**死信出现后** — 不要直接删除记录或清空 outbox。先确认该事件是否已在后端生效，再根据审核记录、固定回执和审计记录选择修复流程。当前管理入口不提供一键重放或批量清理按钮，避免在故障期间扩大影响。

## 🧪 恢复演练

在维护窗口或隔离环境中演练。准备可恢复的数据库备份和无敏感信息的测试事件后，按以下顺序检查：

1. 停止测试 worker，确认检查命令返回 `WORKER_HEARTBEAT_STALE`
2. 恢复 worker，确认健康检查和心跳恢复，再确认没有产生新的重复事件
3. 模拟一个可恢复的后端故障，确认事件先进入待重试状态，依赖恢复后只处理一次
4. 记录演练日期、错误码、恢复时间和验证结果到本地操作记录——不要把现场地址、账号、密钥或日志原文提交到仓库

演练不需要也不应触碰真实用户内容。若必须处理生产死信，先备份并记录当前状态；涉及删除、清理或批量重放时，必须走单独审批。
