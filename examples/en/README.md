# Choose Your Integration Path Here

All files under `examples/` use fictional names and paths — no real accounts, addresses, or keys. Before doing anything else, run the [quickstart](../docs/quickstart.md) local demo first; it automatically validates shared retrieval between two simulated Agents without requiring a hand-generated token hash.

The examples here do only two things: study the HTTP requests of a local prototype, or connect real Codex, Hermes, and OpenClaw instances to an already-deployed shared service. Device registration, credentials, and workspace authorization for a real service are always handled by the admin side; the example files only describe how a client connects to a local Sidecar.

---

## First, Get the Three Names Straight

| Name | What It's For | Example |
|---|---|---|
| Device ID | Distinguishes which machine is making the request | `local-pc` |
| Agent Instance ID | Distinguishes Codex, Hermes, or OpenClaw on this machine | `codex-desktop` |
| Workspace ID | Determines which set of memories a group of Agents can share | `shared-workspace` |

Agents in the same workspace can read each other's authorized shared memories. Do not mix up Device IDs and Agent Instance IDs, and do not rename them arbitrarily.

## Local-Only Experience: SQLite Prototype

No database server or containers required — ideal for testing writes, searches, and permission boundaries first.

1. Generate a random token and save the plaintext value. It is only placed into the current terminal or protected local storage during testing.

   ```powershell
   $token = [guid]::NewGuid().ToString("N")
   $bytes = [Text.Encoding]::UTF8.GetBytes($token)
   $tokenHash = [Convert]::ToHexString([Security.Cryptography.SHA256]::HashData($bytes)).ToLowerInvariant()
   $tokenHash
   ```

2. Copy `principals.example.json` to `principals.local.json`. Create one entry for each Agent you want to connect; replace `token_sha256` with the hash from the previous step, and replace the device, Agent, and workspace names with your own values.

   ```powershell
   Copy-Item .\examples\principals.example.json .\principals.local.json
   ```

   The placeholders in the example are not usable hashes. If even a single character is missing, uppercase letters are used, or placeholder text remains, the Gateway will refuse to start.

3. Start the Gateway and keep this window open.

   ```powershell
   memory-gateway --host 127.0.0.1 --port 8787 --db .\memory.db --principals-file .\principals.local.json
   ```

4. Choose the [OpenClaw HTTP example](openclaw-http.md) to verify a write and read, or proceed to configure the MCP entry points for Codex and Hermes.

Do **not** commit `principals.local.json`, `memory.db`, or test tokens to Git.

## Connecting to a Deployed Shared Service: Local Setup via the Installation Wizard

Production environments do not use SQLite's `principals.local.json`. After an administrator creates a one-time pairing code, run the following on the client machine:

```powershell
.\scripts\setup-shared-memory.ps1 `
  -Mode device `
  -GatewayUrl "https://memory-gateway.example.internal" `
  -DeviceId "local-pc" `
  -DefaultWorkspace "shared-workspace" `
  -Agent @(
    "codex-desktop|codex|Codex Desktop"
    "hermes-desktop|hermes|Hermes Desktop"
  ) `
  -InstallAutostart
```

The three fields of `-Agent` are, in order: Agent Instance ID, type, and display name. The wizard hides the pairing code input, generates a local private key and Sidecar key, saves refresh credentials, and starts a Sidecar that only listens on the loopback address. It also produces fully substituted MCP JSON files; if a key, scheduled task, or JSON file already exists, it stops without overwriting anything.

If pairing succeeded but the machine was interrupted later, continue with the same command plus `-UseExistingCredential`. This switch requires the original device private key to still be present; it only reuses the existing Windows credentials. It never writes credentials back to the terminal or to a config file.

If the Gateway uses an internal CA, place the CA certificate in a protected local location and add `-GatewayCaCertificate "<local CA certificate path>"`. Do **not** disable certificate validation to work around configuration issues.

## MCP Configuration for Codex and Hermes

| User | File to Copy | Two Things to Change |
|---|---|---|
| Codex | `%LOCALAPPDATA%\memory-gateway\mcp\codex-desktop-mcp.json` (generated by the installation wizard) | Import directly or copy the content; no credentials to fill in. |
| Hermes | `%LOCALAPPDATA%\memory-gateway\mcp\hermes-desktop-mcp.json` (generated by the installation wizard) | Import directly or copy the content; no credentials to fill in. |

If you need to manually configure or inspect the fields, you can still refer to the [Codex example](codex-mcp.json) and [Hermes example](hermes-mcp.json). Both JSON files only describe how to start the MCP bridge process. After importing, restart the corresponding Agent and call `memory_sync_status`. If the Sidecar is not running, the key file is missing, or the Agent Instance ID has not been registered, the MCP will report a clear error; fix these prerequisites first — do **not** embed keys into the JSON.

`DefaultWorkspace` must match the value used when starting the Sidecar exactly. When a tool does not pass `workspace_id`, this value is the workspace used for the request; it cannot be a placeholder like `default`.

Agents running inside Docker use the same workspace and tool protocol — no need to copy Windows examples. See [Unified Access for Containerized Agents](../docs/container-sidecar.md): the Bridge's MCP address is always `http://127.0.0.1:8767/mcp` inside the container, and credentials stay in a protected state directory.

[openclaw-http.md](openclaw-http.md) provides the complete local prototype requests, required fields, and common error troubleshooting.
